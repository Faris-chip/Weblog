## 前言

这几年，大模型发展得非常快，参数规模越来越大，训练成本和推理成本也一路抬升。模型变强了，但随之而来的问题也很现实： **显存不够、训练太慢、推理太贵、部署太难** 。因此，“如何加速大模型”逐渐变成了一个绕不开的话题。

在讨论大模型加速时，我们经常会看到一些高频词： **剪枝、量化、蒸馏、稀疏、混合精度、算子优化、并行训练** 。这些技术常常被放在一起讨论，但它们其实并不处在同一个层面，解决的问题也并不完全一样。有的在改模型结构，有的在改数值表示，有的在改执行方式，还有的是在解决训练系统扩展性的问题。

这篇文章就尝试把这几类技术放在一个统一的视角下梳理清楚，重点说明三件事：**它们分别是什么、彼此有什么区别、以及它们在大模型加速中各自扮演什么角色。**

## 大模型加速到底在加速什么

“大模型加速”这个说法听起来很宽泛，但本质上，它无非是在解决下面几类问题：

第一， **模型太大，放不进显存** 。

第二， **计算量太大，训练和推理都太慢** 。

第三， **带宽和访存成为瓶颈，硬件算力发挥不出来** 。

从这个角度看，不同技术的着力点其实很不一样：

* **剪枝、稀疏** ，更关注能不能少算一些内容；
* **量化、混合精度** ，更关注能不能用更低成本的数值表示去算；
* **蒸馏** ，更像是直接换一个更小、更轻的模型；
* **算子优化** ，是在模型不变的前提下，让底层执行更高效；
* **并行训练** ，主要解决的是“大模型怎么训得动、训得快”的问题。

所以，这些方法不是互斥关系。工程上更常见的情况是：**根据瓶颈，把多种技术叠加起来使用。**

## 一、剪枝：删掉冗余部分，让模型变轻

### 1. 什么是剪枝

剪枝，简单来说，就是把模型里那些“不太重要”的参数或者结构删掉，让模型变得更小、更省计算。

这个思路其实并不复杂。一个训练完成的模型里，并不是每个参数都同等重要，很多连接、通道、注意力头，甚至某些层，本身就存在一定冗余。剪枝的目标，就是尽可能识别这些冗余部分，并把它们去掉。

### 2. 剪枝通常怎么做

剪枝可以分成两大类。

一种是 **非结构化剪枝** 。它会直接把单个权重置零，比如把一部分数值很小、贡献有限的参数删掉。这种方法从参数数量上看效果很明显，但问题在于，很多硬件和计算库并不会因为矩阵里多了很多 0 就自动跳过计算，因此它常常“理论压缩明显，实际加速有限”。

另一种是 **结构化剪枝** 。它不是删零散的权重，而是按更规整的单位删除，比如删整条通道、删整个 attention head、删一部分 FFN 神经元，甚至删掉整层。结构化剪枝的优点是更符合硬件执行习惯，因此更容易带来真实的推理加速。

此外，剪枝还可以分成一次性剪枝和迭代式剪枝。一次性删太多，模型容易掉点；迭代式通常更稳一些，因为每轮删一点，再做微调，性能损失相对可控。

### 3. 它对大模型加速有什么作用

剪枝的价值主要体现在三个方面。

首先，它能减少模型参数量，从而降低存储开销。

其次，如果剪得足够规整，它可以减少实际 FLOPs。

最后，在部署场景中，它有机会降低推理延迟。

不过这里有个很现实的问题：**剪枝能不能真正提速，不光取决于删了多少，还取决于删得是否“规整”，以及底层框架和硬件是否支持。**

### 4. 它适合什么场景

剪枝比较适合已经训练好的模型，希望进一步压缩和部署优化的场景。尤其是在边缘设备、资源受限平台、或者对模型规模比较敏感的部署环境中，剪枝会比较有吸引力。

在大模型里，常见做法往往不是特别激进地删整个网络，而是更温和地处理，比如减少 FFN 宽度、减少 attention head 数量，或者对层数做适当裁剪。

## 二、量化：用更少的 bit 表示模型

### 1. 什么是量化

量化，就是把模型中原本用高精度表示的数，改成更低精度的形式来存储和计算。比如原来使用 FP32、FP16，现在改成 INT8、INT4，或者更激进一点，用 FP8。直观理解就是：**原来一个数要占 16 位、32 位，现在只让它占 8 位、4 位。**

### 2. 量化为什么很重要

量化之所以在大模型部署中这么常见，是因为它直接击中了两个核心瓶颈：**显存容量**和 **显存带宽** 。

模型越大，参数越多，显存越容易不够用；而推理过程中，很多时候并不只是“算得慢”，更是“搬数据太慢”。量化以后，每个参数占用空间变小，搬运的数据量也变少，带宽压力自然下降。

如果硬件本身对低比特运算支持得比较好，比如支持 INT8、FP8 Tensor Core，那么量化还能带来实打实的吞吐提升。

### 3. 常见量化方法

从流程上看，量化通常分成两类：

一种是 **后训练量化** ，也就是模型训练结束后再去量化。它的优点是简单，部署方便，不需要重新训练；缺点是位宽降得太低时，精度损失可能比较明显。

另一种是 **量化感知训练** ，在训练阶段就把量化误差考虑进去。这种做法更复杂，但通常能在低比特下保住更好的精度。

从对象上看，量化也不只是量化权重，还可以量化激活、KV cache，甚至训练中的梯度。

### 4. 它对大模型加速的作用

在大模型推理中，量化往往是最实用的技术之一。它既能缩小模型体积，也能缓解访存压力，还能提高硬件吞吐。

尤其是对长上下文推理来说，KV cache 往往会随着序列长度快速膨胀，此时做 KV cache 量化，收益会非常明显。

所以从部署角度看，量化不是一个“可有可无”的优化项，而常常是第一批要考虑的技术。

### 5. 它适合什么场景

量化特别适合推理部署场景，尤其是：

* GPU 显存紧张；
* 模型规模较大；
* 需要提升吞吐；
* 希望降低单次推理成本；
* 本地大模型部署或者边缘部署。

很多开源大模型之所以能在消费级显卡上跑起来，量化往往是关键原因之一。

## 三、蒸馏：把大模型的能力“压”进小模型

### 1. 什么是蒸馏

蒸馏的核心思想不是直接改原模型，而是让一个小模型去学习大模型的行为。也就是说，大模型当老师，小模型当学生，让学生尽可能模仿老师的输出和能力。

学生模型学习的对象不一定只是最终答案，还可以是中间层特征、输出概率分布、注意力模式等等。

### 2. 蒸馏为什么能带来加速

蒸馏本身不是推理时的实时优化动作，它更像是一种“模型替换”策略。

也就是说，蒸馏真正带来的加速，不是让原始大模型跑得更快，而是让你最后根本不再部署原来的大模型，而是部署一个更小、更快的学生模型。

这和剪枝、量化的逻辑不太一样。剪枝和量化是在给原模型减负，而蒸馏更像是重新造一辆更轻的车。

### 3. 常见蒸馏方式

比较基础的是 logits 蒸馏，也就是让学生模型去拟合老师的输出分布。除此之外，也有特征蒸馏、注意力蒸馏、任务蒸馏等。

在大模型时代，指令蒸馏和响应蒸馏也很常见。具体来说，就是让大模型先生成高质量数据，再用这些数据训练一个小模型，让它继承大模型在某一类任务上的能力。

### 4. 它对大模型加速的作用

蒸馏最适合的场景是：原始大模型能力很强，但部署成本太高。此时，与其强行压缩原模型，不如直接训练一个更小的模型，让它尽量学会大模型在特定任务上的能力。

因此，蒸馏在工程中的意义往往不是“让最强模型更快”，而是“让一个更小的模型达到够用的效果”。

### 5. 它适合什么场景

蒸馏很适合以下场景：

* 任务目标比较固定；
* 不需要一个通用全能的大模型；
* 更关心低延迟、高 QPS、低部署成本；
* 希望把云端能力迁移到边缘端或本地端。

比如，把通用大模型蒸馏成客服问答模型、代码补全模型、分类模型、排序模型，这些都很常见。

## 四、稀疏：不是所有东西都值得每次都算

### 1. 什么是稀疏

稀疏这个概念比剪枝更宽。它的核心不是“删掉了什么”，而是：**在参数、激活或者计算路径中，有很多部分本来就不需要参与计算。**

稀疏可以出现在很多地方，比如权重稀疏、激活稀疏、注意力稀疏，甚至模型结构级别的条件稀疏。

### 2. 稀疏有哪些典型形式

最容易理解的是权重稀疏，也就是参数矩阵里很多元素为 0。

其次是激活稀疏，也就是一层前向传播时，很多神经元实际输出为 0。

再进一步，就是注意力稀疏：并不是每个 token 都需要和所有 token 两两计算 attention。

还有一种很重要的形式是 **条件稀疏** ，比如 MoE。MoE 模型虽然总参数量很大，但每个 token 实际只激活少量专家，因此单次计算量仍然可控。

### 3. 稀疏为什么能加速

从理论上说，稀疏当然能减少计算量，因为有很多内容可以直接跳过。但从工程上看，稀疏并不天然等于高性能。原因很简单：硬件喜欢规则、连续、密集的计算。

如果稀疏模式太零散，就会带来额外索引开销、访存不连续、并行度下降等问题。结果就是，理论 FLOPs 降了不少，实际速度却未必有明显提升。

因此，稀疏要真正变成加速收益，往往需要满足两个条件：

一是稀疏模式尽量规则；

二是底层硬件和计算库要有对应支持。

### 4. 稀疏在大模型里的价值

当前大模型里，最成功的稀疏路线之一就是  **MoE** 。MoE 的价值就在于：模型总参数可以很大，但每个 token 并不会激活全部参数，而是只走一小部分专家。这实际上是在不显著增加单 token 计算量的前提下，提升模型容量。

另外，在长上下文处理中，注意力稀疏也是一个很自然的方向。因为序列一长，全连接 attention 的代价会非常高，此时让 attention 只保留局部窗口或少量全局连接，往往更合理。

### 5. 它和剪枝的区别

剪枝和稀疏经常被混用，但它们的重点并不一样。

剪枝更强调“主动删减冗余部分”；

稀疏更强调“计算过程中本来就有很多无效或可跳过的部分”。

可以说，剪枝经常会产生稀疏，但稀疏不一定来自剪枝。MoE 就是一个很典型的例子，它的稀疏不是通过传统剪枝得来的，而是通过条件路由机制实现的。

## 五、混合精度：该省的地方省，该稳的地方稳

### 1. 什么是混合精度

混合精度的意思是，在同一次训练或推理过程中，不是所有数值都统一使用一种精度，而是根据不同环节的需求，选择不同的精度表示。

典型组合包括 FP32、FP16、BF16，更新一些的系统里也会用到 FP8。

### 2. 为什么不直接全都低精度

原因在于，不同运算对数值精度的敏感程度不同。

有些部分可以放心降精度，比如大量矩阵乘；但也有些部分对数值稳定性更敏感，比如梯度累计、归一化、loss 计算等。

所以混合精度的核心思路不是“盲目降精度”，而是：**在大部分可以省的地方使用低精度，在关键位置保留更高精度。**

### 3. 它对大模型加速的作用

混合精度在训练中尤其重要。因为训练阶段不仅要做前向传播，还要做反向传播、梯度保存、参数更新，对显存和计算吞吐的要求都更高。

如果大量张量从 FP32 换成 FP16 或 BF16，那么显存占用会明显下降，同样的显存预算下可以容纳更大 batch，整体吞吐也通常会提升。

在推理阶段，混合精度同样有价值，特别是在支持 BF16 或 FP8 的硬件平台上。

### 4. 它适合什么场景

混合精度几乎是现代大模型训练中的“标配”之一。只要你是在多卡 GPU 上训练大模型，混合精度通常都会是默认方案的一部分。

尤其是在：

* 显存有限；
* 希望提升训练吞吐；
* 硬件支持 Tensor Core、BF16、FP8；

  这些条件下，它的收益非常明显。

### 5. 它和量化的区别

量化往往更偏向部署压缩和推理优化，目标是用更低 bit 做更便宜的推理。而混合精度则更偏向训练过程中的数值-性能折中，重点是“尽量快，同时不把训练搞崩”。

因此，虽然两者都和低精度相关，但一个更偏“压缩部署”，一个更偏“训练友好”。

## 六、算子优化：模型不变，但执行方式变快

### 1. 什么是算子优化

一个模型最终落到硬件上，并不是以“Transformer”或者“神经网络层”这种抽象概念执行的，而是被拆解成一个个具体的算子，例如矩阵乘、Softmax、LayerNorm、Attention、激活函数等。

算子优化，指的就是对这些底层算子的实现方式进行优化，让同样的数学运算在硬件上执行得更快。

### 2. 为什么算子优化很关键

很多时候，模型理论 FLOPs 并没有变，但只要把关键算子优化好了，实际速度就会提升很多。原因在于，真实系统中的瓶颈并不只是算术操作本身，还包括访存、中间结果回写、kernel 启动开销、张量布局变换、同步等待等一系列问题。

也就是说，**同样一个模型，写法不同、内核不同、执行调度不同，跑出来的速度可能差很多。**

### 3. 常见优化方式

一个很常见的思路是 **算子融合** 。比如把多个连续的小算子合成一个 kernel，减少中间结果落显存，降低 kernel 启动开销。

另一个重点方向是优化 Attention。大模型推理中的热点通常就集中在 QKV 投影、Attention 计算、FFN 大矩阵乘这些模块里，因此像 FlashAttention 这样的优化非常有代表性，本质上是在减少不必要的中间存储和访存。

此外，还包括张量布局优化、tiling、并行切分、小算子合并等。这些做法不会改变模型结果，但会让硬件利用率更高。

### 4. 它对大模型加速的作用

算子优化的一个重要特点是：**它不一定改模型，但经常能显著改性能。**

尤其是在部署侧，一个系统是否能把 GPU 或 NPU 跑满，往往就取决于这些热点算子的实现质量。很多高性能推理框架的核心竞争力，实际上就在这里。

## 七、并行训练：不是让模型变小，而是让模型训得动

### 1. 什么是并行训练

随着模型规模不断增大，训练时经常会遇到一个很现实的问题：单张卡的显存放不下模型，或者即便放得下，训练时间也长得不可接受。并行训练就是为了解决这个问题而出现的。

简单说，就是把训练任务分散到多张卡、多台机器上一起完成。

### 2. 常见并行方式

最基础的是 **数据并行** 。每张卡放一份完整模型，不同卡处理不同 batch，最后同步梯度。这种方式最好理解，也最常见，但前提是单卡本身能放得下模型。

如果模型太大，就要上 **模型并行** 。模型并行又可以继续分成张量并行和流水并行。张量并行是把同一层内部的大矩阵切到多张卡上算；流水并行则是把不同层分布到不同设备上，让激活像流水线一样流过不同设备。

再进一步，还有 **ZeRO** 这类参数分片技术，它的重点不是切计算，而是把参数、梯度、优化器状态分散到不同设备上，以减少冗余存储，降低显存压力。

实际的大模型训练通常不是只用一种，而是 **数据并行、张量并行、流水并行、参数分片一起上** ，形成混合并行。

### 3. 它对大模型加速的作用

并行训练解决的核心问题不是部署成本，而是训练可行性和训练效率。有些模型不是“训得慢”，而是如果不上并行，根本就训不了。

因此，并行训练的第一价值是“让它能训”；第二价值才是“让它训得更快”。

### 4. 它适合什么场景

并行训练主要适用于大模型预训练、继续训练、全参数微调、多机多卡训练等场景。对于参数量不大、训练规模不高的小模型来说，并行训练未必划算，因为通信同步本身也有代价。但一旦进入大模型阶段，并行几乎就是必选项。

## 把这些技术放在一起看：它们到底有什么区别

讲到这里，其实可以把这几种技术做一个更清晰的归类。

### 1. 谁是在“改模型”

* 剪枝
* 量化
* 蒸馏
* 稀疏

这些技术都和模型本身有关，要么在删减模型，要么在压缩表示，要么在构造新的轻量模型。

### 2. 谁是在“改执行方式”

* 算子优化
* 并行训练
* 混合精度（某种意义上也介于两者之间）

这类方法不一定明显改变模型结构，但会显著影响模型实际跑起来的效率。

## 结语

剪枝、量化、蒸馏、稀疏、混合精度、算子优化、并行训练，这些技术看上去都在讲“加速”，但它们其实站在不同层面回答同一个问题：**怎样在有限资源下，把大模型跑得更高效。**

所以，大模型加速从来都不是单点突破，而是模型、算法、系统、硬件共同作用的结果。真正有价值的优化，往往也不是孤立地讨论某一个名词，而是搞清楚：**当前系统的瓶颈到底在哪里，然后用最合适的技术去对症下药。**
