在 2025 年，我作为队员参加了第九届集创赛的竞业达杯，我们当时设计了五级流水线的 RISC-V 处理器，支持基础的 RV32I 指令集，在 Kintex-7 的 FPGA 开发板上成功运行，最高时钟频率为 180 MHz，比赛时稳定运行的时钟频率为 160 MHz；2026 年，我作为队长再次参加集创赛竞业达杯，在我们去年的 CPU 基础上进行进一步的改造。

<!-- more -->

## 去年完成的基础

- 区域赛的 CPU 最高运行频率为 180 MHz，支持 37 条 RV32I 指令；
- 国赛的 CPU 成功移植 RT-Thread Nano 操作系统，支持 RV32IM 指令集，扩展了异常、中断处理等功能，最高运行频率为 125 MHz；
- CPU 为哈佛结构，五级流水线架构，使用静态分支预测，DRAM 和 IROM 使用的是 Distributed RAM IP 核。

## 目前遇到的问题

- 关键路径出现在访存 MEM 阶段，要想进一步提升频率，必须优化关键路径；
- 目前 CPU 使用的是简单的静态分支预测，跳转分支的预测成功率较差，需要替换成动态分支预测；
- 在更换 Distributed RAM 为 Block RAM 的过程中，遇到了一些时序问题。

## 更换 Block RAM 过程

### 为什么要更换

我们最开始写 CPU 的时候，指令存储器和数据存储器用的是 Distributed RAM ，它是用 FPGA 的 LUT 表拼凑出来的，纯组合逻辑，时序性能比较差，优点是，在前期设计 CPU 的时候容易理解读写逻辑，方便快速设计，体现在读数据很直接，传进去读数据的地址，可以在同一个周期内读回数据，所以前面的五级流水设计中，我们在 MEM 访存阶段，执行的读写操作都是在一个周期内就要完成。

但后来我们想把设计真正放到 FPGA 上跑，尤其是想冲更高频率，就发现一个问题，**板子上最合适的片上存储资源不是使用 LUT “拼凑”成的组合读存储器 Distracted RAM，而是 FPGA 上的高性能专用存储器 Block RAM。**

因为 Block RAM 才是 FPGA 真正高效、稳定、资源利用也合理的实现方式，我们如果继续用原来的那种写法，MEM 访存阶段的时序压力太大，成为整个设计的关键路径，所以我们就决定把原来的存储器模型正式替换成 BRAM IP。

### 流水线的拆分

我们真正开始替换之后，问题马上就暴露出来了——**Block RAM 不是我们之前的那种“组合读”行为，它本质上是同步读的。** 也就是说，地址打进去以后，数据不是同一个周期就稳定出来，而是要 **延后一拍，才能拿到数据** 。为此，我们需要拆分流水，将 MEM 访存阶段拆分成 MEM1 和 MEM2 两个阶段，在 MEM1 传入要读数据的地址，在 MEM2 阶段读出对应的数据。

这个改动的逻辑是这样的：

* **MEM1** ：负责把访存请求真正发给 BRAM，比如地址、写使能、写数据这些信号在这一拍送出去；
* **MEM2** ：专门负责在下一拍接收 BRAM 返回的读数据，然后再送去写回或者供后续使用。

这样一来，CPU 的流水定义就和 FPGA 上 BRAM 的真实时序对齐了，不再是拿旧模型去硬套新存储器。

### 出现的问题

#### 1、数据相关和冒险检测

针对新的数据冒险，我们按照之前解决五级流水的数据冒险的思路，在 Harzad_Detection 模块中，增加新的数据依赖检测机制和数据前递通路，从而成功解决拆分流水引入的新的数据冒险。

真正比较难理解的是 load-use 型冒险，我们可以理解成一种无法通过数据前递解决的数据冒险，其实，这个问题本质上是，load 指令的数据，什么时候才真正可用？

我们举个例子来理解，在我们原来的五级流水线中，假如我们的 load 指令取出的数据，在后续的一条指令 add 中就要使用，这时就产生了数据冒险，而且这种冒险无法通过数据前递消除，必须使用 stall 停顿机制，暂停 IF / ID / EX 之间的级间寄存器，同时 flush 冲刷掉 EX 和 MEM 之间的级间寄存器，防止错误数据向后传递。

```assembly
lw   x1, 0(x2)
add  x3, x1, x4
sub  x5, x1, x6
```

在后来改成六级流水之后，load 指令取出的数据，如果在后续的两条指令 add 和 sub 就要使用(可以是其中一条指令，也可以是两条指令)，这时候的 load-use 型冒险就不能再按原来的五级思路理解了，这是因为：

* `lw` 在 EX 才算出地址
* 在 MEM1 才真正把地址送给 BRAM
* 在 MEM2 才真正拿到读出的数据

所以对后面的 `add`、`sub` 来说，`x1` 在它们进入 EX 的时候，很可能还没有准备好。这就产生了新的 load-use hazard，而且比原先更“长”：

* 原来可能只需要盯住“紧跟着的一条”
* 现在的六级流水里，**紧跟的第 2 条、第 3 条** 都可能落在危险区间里

换成 BRAM 以后，冒险检测要重新定义“load 结果何时可转发”，以前默认 MEM 阶段末就能转发，但现在必须明确，**load 的读数据只有在 MEM2 之后才可靠。** forwarding 条件不能再沿用原来的判断，不然就会把一个“还没出来的数据”错误地当成可用值。

另外，stall 条件要向前扩展，以前只拦住紧邻 load 的那条相关指令，现在因为数据回来更晚，所以 hazard detection unit 必须识别更长的危险窗口，检测逻辑要从原来的：

* “EX/MEM 里是不是有一个 load，ID 里这条是不是在用它的 rd”

扩展成更贴合六级流水的形式：

* “前面处在 EX、MEM1、MEM2 的 load，哪一级的数据还没真正 ready”
* “当前 ID / EX 的指令是否正要读取这个 rd”
* “如果读取发生在结果 ready 之前，就必须停顿”

#### 2、外设行为同步

因为比赛这个缘故，我一直在关注 CPU 核内在的运行逻辑，完全没有意识到外设在改造六级流水的过程重要性，结果在调试的时候遇到了很多奇奇怪怪的问题，后面才发现是外设的读写逻辑和我们的 BRAM 不一致导致的冲突，我们**不能只让 CPU 自己适应同步 BRAM，还得让它和外设、总线、存储映射接口的行为认知一致。**

那么为什么会有这个问题，这还要追溯到我们最开始使用的存储器 Distributed RAM，它是纯组合逻辑，我们的所有外设还有总线桥的地址转发逻辑也是按照组合逻辑的读写行为设计的，CPU 对外部世界形成的是一种组合逻辑认知：

* 地址一给，数据很快就到
* 写使能打一拍，外面立刻就像已经完成了
* 读写的响应和 CPU 的节拍几乎天然对齐

但在更换成时序逻辑的 BRAM 之后，我们忽略了其他外设还是组合逻辑的事实，导致了一系列的地址和读写数据冲突，最后我们通过人为暂停一拍的方式，将其他外设也改造成了时序逻辑，解决了上述问题。

## 改造感受

这次把原来的Distributed RAM 换成 Block RAM，对我来说其实不是一次单纯的模块替换，更像是一次很深的“系统认知修正”。

一开始我以为问题主要出在存储器本身，后来才慢慢发现，真正被改变的是整个 CPU 对访存时序的理解。以前在仿真模型下很多想当然成立的事情，到了真实 FPGA 环境里就不再成立了，比如数据什么时候返回、相关指令什么时候该停、错误路径上的指令怎么清掉，这些都必须重新想清楚。这个过程虽然比较折腾，也走了不少弯路，但反过来让我对流水线、冒险处理和 SoC 真实时序有了更扎实的理解。

现在再回头看，这段经历挺有价值的，因为它让我从“能跑仿真”进一步走到了“更接近真实硬件实现”的层面。
